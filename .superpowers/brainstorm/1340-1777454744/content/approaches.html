Implementation Approach

Three ways to structure the refactor — pick the one that fits how you want to ship.

A

Full Modular Refactor Recommended

Break the ExamsHeroCard into all the components the spec asks for. Each component is focused, reusable, and reads from context directly.

src/components/dashboard/exams/
├── ExamCard.jsx ← replaces ExamsHeroCard
├── WeakTopicsSection.jsx ← refactored with subject filter
├── WeakTopicCard.jsx ← new: individual topic card
├── EmptyState.jsx ← new: reusable empty states
├── StudySuggestion.jsx ← new: "Focus Today" strip
├── StudyPlanModal.jsx ← new: day-by-day calendar modal
├── ExamCountdownSection.jsx ← updated urgency thresholds
└── AddExamModal.jsx ← updated with subject dropdown

Pros

  • Clean separation of concerns
  • Easy to extend later
  • Matches spec exactly

Cons

  • More files to create
  • Slightly more upfront work
B

Patch-in-place

Keep the existing file structure. Modify ExamsHeroCard.jsx and WeakTopicsSection.jsx directly. Add new features as sections inside those files.

Pros

  • Fewer files touched
  • Lower risk of regression

Cons

  • ExamsHeroCard grows into a god component
  • Hard to test individual pieces
  • Doesn't match spec structure
C

Fix Core Bug Now, Features Later

Ship only the subject-filtering fix and conditional empty states first. Add Study Suggestion, Study Plan, and integrations as a separate pass.

Pros

  • Fastest to ship something working
  • Lower risk

Cons

  • Two implementation passes
  • Students miss the smart features longer

⚡ All approaches include: subject dropdown in AddExamModal, DB schema change, sessionStorage quiz linking, and day-by-day study plan calendar.
